This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Feedme — a teaching-demo cookbook app (Phase I of the brief). npm-workspaces monorepo: `server/` (Express API), `client/` (TanStack Start SPA), `openapi/` (hand-written API contract). Built to show juniors a "regular" React app; keep it simple and idiomatic over clever.

## Commands

Run from the repo root (workspace scripts fan out to `server`/`client`):

```bash
npm install                # install both workspaces
npm run dev                # API :8080 + client :3000 together (concurrently)
npm run dev:server         # API only
npm run dev:client         # client only (Vite proxies /api -> :8080)
npm run typecheck          # tsc --noEmit for both packages — the main correctness gate
npm run build              # server (tsc) + client (vite build, incl. SPA prerender)
npm run openapi:lint       # redocly lint openapi/openapi.yaml
```

There is **no test runner and no linter (ESLint)** configured — `typecheck` is the only automated check. Verify behavior by running the app. Node isn't on the default PATH here; it comes from nvm and is symlinked into `~/.local/bin` (node/npm/npx). `esbuild`'s install script must be allowed (`allowScripts` in root `package.json`) or `tsx`/Vite break — run `npm rebuild esbuild -w <pkg>` if a fresh install skips it.

## Architecture

**Client ⇄ server contract.** Client always calls `/api/...` (Vite dev proxy → Express on :8080). Keep three things in sync when changing an endpoint's shape: the route in `server/src/routes/`, the response shaper in `server/src/serializers.ts`, and the client DTO in `client/src/lib/types.ts`. `openapi/openapi.yaml` is authored by hand — update it too. The server serves the raw spec at `/api/openapi.yaml` and interactive Swagger UI at `/api/docs` (an HTML page in `index.ts` loading Swagger UI from a CDN — no dep).

**Server is stateful in memory, no DB.** `server/src/data/store.ts` exports mutable arrays (`users`, `recipes`, `cookbooks`, `comments`) seeded from `data/*.ts`; routes mutate them directly and everything resets on restart. Bcrypt hashes are computed at boot from plaintext seed passwords. `nextId(prefix)` generates ids.

**Serializers, not raw entities.** Routes return `serialize*(entity, viewerId)` output — author inlined, `likesCount`/`commentsCount` computed, and `likedByMe` is relative to the requesting user. Never return stored entities directly (they contain `passwordHash`, `likedBy`, etc.).

**Auth (httpOnly cookie).** On login/register the server issues the JWT as an **httpOnly cookie** `feedme_token` (`server/src/auth/cookie.ts`; options in `config.ts`) — never in the response body, so client-side JS can't read the token. `requireAuth` (`server/src/auth/middleware.ts`) reads the token from that cookie (falling back to an `Authorization: Bearer` header so Swagger/curl still work) and attaches `req.user`. Applied at the router level for `/me`, `/recipes`, `/cookbooks`, `/comments`, so those handlers assume `req.user!`; only `/auth/*` is public. `POST /auth/logout` clears the cookie. Ownership is enforced per-handler (403 on non-owner PATCH/DELETE). The client sends the cookie via `credentials: "include"` in `lib/api.ts`; `cors` is configured with `credentials: true`.

**Validation is doubled.** Zod schemas exist on both sides. Server: `server/src/validation/schemas.ts` + `validateBody(schema)` middleware (returns `{ message, errors: {field: msg} }` on 400). Client: `client/src/lib/schemas.ts` used via `zodResolver` in react-hook-form. Changing a rule usually means editing both.

**Client rendering & routing.** SPA mode (`tanstackStart({ spa: { enabled: true } })` in `vite.config.ts`) — no meaningful SSR. `client/src/router.tsx` (`getRouter`) wires one QueryClient into router context via `setupRouterSsrQueryIntegration`. File-based routes in `client/src/routes/` use pathless layouts: `_authLayout.*` (redirects away if logged in) and `_appLayout.*` (guards + renders Header/Footer). Because the JWT cookie is httpOnly (unreadable by JS), route guards can't check auth synchronously — `beforeLoad` is **async** and calls `context.queryClient.ensureQueryData(meQueryOptions)` (a cached `GET /me`, defined in `client/src/lib/auth.ts`), redirecting on 401. `AuthContext` shares that same `['me']` query, so `login()`/`setUser()` seed it and `logout()` clears the cache. Dotted filenames = nesting (e.g. `_appLayout.profile.cookbooks.tsx` → `/profile/cookbooks`). `src/routeTree.gen.ts` is auto-generated by Vite and gitignored — never edit it.

**Data flow on the client.** Reads go through per-resource TanStack Query hooks in `client/src/hooks/` (`useRecipes`, `useCookbooks`, `useComments`, `useProfile`); mutations invalidate the matching query keys so lists refresh. All HTTP goes through `client/src/lib/api.ts` `request()`, which sends `credentials: "include"` (for the auth cookie) and throws `ApiError` (carrying `errors` field map) on non-2xx. Auth state is a single React context (`client/src/context/AuthContext.tsx`, `useAuth`) backed by the `['me']` query.

**URL-driven UI.** Search pages and detail/create modals are controlled by typed search params (`validateSearch` with Zod), e.g. `/cookbooks?cookbook=c1` opens the detail modal, `/profile/recipes?create=true` opens the create form. When patching search params prefer building from the route's own typed `Route.useSearch()` value (`search: { ...params, ...next }`) rather than the functional `(prev) => …` updater — `prev` is the global search union across all routes and causes type errors.

**Styling.** Plain CSS, one `.css` file imported per component. All colors/fonts/shadows come from CSS custom properties in `client/src/styles/tokens.css`; use the tokens, don't hard-code values.

**Images.** Seed food photos are Unsplash CDN URLs by photo id (`img(id)` helpers in `data/recipes.ts`/`cookbooks.ts`); avatars from pravatar.cc. loremflickr was tried and abandoned (renders blank under concurrent load) — don't reintroduce it.
